User Explorer

Aplicación web en JavaScript vanilla que consume la API pública de jsonplaceholder, valida los datos recibidos y los presenta con un diseño oscuro moderno.

Vanilla JS Bootstrap 5 22 tests · Jest 94% cobertura GitHub Pages Sin bundler

Introducción

User Explorer es una aplicación frontend minimalista que demuestra buenas prácticas de desarrollo sin necesidad de un framework ni una herramienta de build: módulos ES nativos en el navegador, validación de datos en capa de API, manejo explícito de errores en UI y una suite de tests unitarios con Jest.

Tecnologías

Frontend

  • HTML5 + ES Modules (sin transpilación en navegador)
  • Bootstrap 5.3 — layout, tabla, modal (CDN)
  • FontAwesome 6.5 — iconografía (CDN)

Testing

  • Jest 29 — test runner
  • babel-jest + @babel/preset-env — soporte ESM en Node
  • Mocks de fetch con jest.fn()

Arquitectura

La aplicación sigue un flujo unidireccional de tres capas:

UI Layer
app.js / index.html
API Layer
api.js
Validation Layer
validator.js
Usuario pulsa botón → app.js → fetchUsers() → fetch(API) → validateUser() → renderUsers()

Principios de diseño

  • Separación de responsabilidades — cada módulo tiene un único propósito.
  • Fail fast — los datos se validan antes de llegar a la UI; nunca se muestran datos parciales.
  • Error surfacing — todos los errores de red, servidor y validación se presentan al usuario de forma descriptiva.
  • Sin efectos secundarios silenciosos — los usuarios inválidos descartados se registran en consola con console.warn.
  • Sin estado global mutable — la UI se reconstruye completamente con cada carga.

validator.js

Módulo de validación puro, sin efectos secundarios ni dependencias externas.

validateUser(user)

validateUser(user: object | null | undefined): { valid: boolean, errors: string[] }

Verifica que el objeto usuario cumpla todos los campos mínimos requeridos por la UI.

ParámetroTipoDescripción
user object | null | undefined El objeto usuario a validar. Se acepta cualquier valor para devolver un error controlado.
@returns { valid: boolean, errors: string[] } — valid es true solo si el array errors está vacío.

Reglas de validación

CampoRegla
namestring no vacío (trim)
emailstring no vacío + regex /^[^\s@]+@[^\s@]+\.[^\s@]+$/
address.citystring no vacío (trim)
address.geo.latpresente, no null, no string vacío
address.geo.lngpresente, no null, no string vacío

Ejemplo

import { validateUser } from './src/validator.js';

const result = validateUser({
  name: 'Ada Lovelace',
  email: 'ada@example.com',
  address: { city: 'London', geo: { lat: '51.5', lng: '-0.1' } }
});
// → { valid: true, errors: [] }

const bad = validateUser({ name: '', email: 'not-an-email' });
// → {
//     valid: false,
//     errors: [
//       'name is required and must be a non-empty string',
//       'email format is invalid',
//       'address is required'
//     ]
//   }

api.js

Capa de acceso a datos. Envuelve fetch con manejo explícito de errores de red, status HTTP y JSON malformado. Delega la validación semántica a validator.js.

fetchUsers()

fetchUsers(): Promise<object[]>

Llama a GET /users, parsea la respuesta y filtra los usuarios inválidos. Si ningún usuario pasa la validación, lanza un error con el detalle completo.

ParámetroTipoDescripción
—voidSin parámetros.
@returns Promise<object[]> — Array de usuarios que superaron la validación.
@throws Error si: red caída · status HTTP no-2xx · cuerpo vacío · JSON malformado · respuesta no es array · todos los usuarios son inválidos.
Comportamiento parcial Si algunos usuarios son inválidos se devuelven solo los válidos. Los descartados se notifican via console.warn con el detalle de errores.

fetchUserById(id)

fetchUserById(id: number | string): Promise<object>

Llama a GET /users/:id y valida el objeto devuelto.

ParámetroTipoDescripción
id number | string Identificador del usuario. Requerido; lanza error si está vacío/null/undefined.
@returns Promise<object> — Objeto usuario validado.
@throws Error si: id ausente · red caída · status no-2xx · JSON inválido · usuario no pasa validación.

Ejemplo

import { fetchUsers, fetchUserById } from './src/api.js';

// Cargar listado
const users = await fetchUsers();
console.log(users.length); // hasta 10

// Cargar detalle
const user = await fetchUserById(3);
console.log(user.address.geo); // { lat: '-68.6102', lng: '-47.0653' }

// Manejo de error
try {
  await fetchUserById(999);
} catch (err) {
  console.error(err.message); // Server responded with status 404 Not Found
}

app.js

Orquesta la interacción entre el DOM y la capa API. No contiene lógica de negocio ni acceso directo a red; toda esa responsabilidad está delegada a api.js.

Responsabilidades

  • Escucha el click de "Cargar usuarios" → llama fetchUsers() → llama renderUsers().
  • Escucha clicks en los botones de ojo de la tabla (delegación de eventos) → llama fetchUserById(id) → abre el modal Bootstrap.
  • Muestra spinner durante las peticiones y deshabilita el botón para evitar llamadas duplicadas.
  • En caso de error, muestra un mensaje descriptivo y limpia la tabla para evitar estado inconsistente.
  • Escapa todo el HTML interpolado con escapeHtml() para prevenir XSS.
  • Genera el enlace de Google Maps con las coordenadas geo.lat y geo.lng del usuario.

Seguridad: prevención de XSS

Todos los datos externos se pasan por escapeHtml() antes de interpolarse en el DOM. Esta función escapa los caracteres & < > " '.

Quick Start

1. Instalar dependencias de desarrollo

npm install

2. Ejecutar la aplicación en local

El proyecto usa módulos ES nativos, por lo que necesita un servidor HTTP (no file://): 

# Python 3
python3 -m http.server 8080

# Node.js (npx)
npx serve .

# Visita → http://localhost:8080

3. Ejecutar los tests

# Modo interactivo (watch)
npm test

# CI — una pasada con cobertura
npm run test:ci

Despliegue en GitHub Pages

El proyecto no requiere build step, por lo que el despliegue es directo:

  1. Sube el repositorio a GitHub.
  2. Ve a Settings → Pages.
  3. En Source, selecciona la rama main y la carpeta / (root).
  4. GitHub publicará en https://<usuario>.github.io/<repo>/.
  5. La documentación estará disponible en /docs/ si activas la opción correspondiente.
Nota node_modules/ y coverage/ deben estar en .gitignore para no publicarlos.
